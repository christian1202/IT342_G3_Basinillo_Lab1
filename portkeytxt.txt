EXECUTIVE SUMMARY
1.1 Project Overview
PortKey is an Intelligent Customs Clearance Platform designed primarily for Brokers. It serves as a focused tool to manage shipments, track cargo lifecycles, and monitor a real-time demurrage countdown to prevent costly port penalties. The system utilizes a custom-built Spring Boot backend, a Next.js web frontend, and a native Kotlin Android application.
1.2 Objectives
1. Develop a fully functional shipment tracking and demurrage management system for brokers.
2. Implement a secure, custom JWT authentication system with Google OAuth2 integration, bypassing third-party Backend-as-a-Service providers.
3. Ensure real-time urgency updates via WebSockets and automated alerts via email and Android push notifications.
4. Deploy a cohesive three-tier architecture with a Next.js web client and an XML-based Kotlin mobile application.
1.3 Scope
Included Features:
* Custom user registration and authentication (BCrypt, custom JWT).
* Shipment creation and lifecycle tracking (Arrived to Released).
* Real-time Demurrage Doomsday Clock with visual urgency indicators.
* Document Vault via Cloudflare R2 for secure file uploads.
* ExchangeRate-API integration for automated PHP duty conversions.
* Responsive, modern Next.js web interface.
Excluded Features:
* Clerk or any Backend-as-a-Service auth.
* AI features including Smart-Scan OCR and Compliance Copilot (reserved for future phases).
* Freight booking and post-release warehouse management.


* Automated SMS or push notifications.
* Third-party accounting software integrations (e.g., QuickBooks, Xero).

2.0 FUNCTIONAL REQUIREMENTS SPECIFICATION
2.1 Project Overview
Project Name: PortKey
Domain: Logistics / Customs Brokerage
Primary Users: Customs Broker / Logistics Specialist
Problem Statement: Brokers need a centralized, real-time tool to track incoming container shipments and avoid devastating demurrage charges due to missed deadlines.
Solution: A centralized digital ledger for customs brokers. It provides a structured way to encode shipment details, and track the clearance lifecycle from a single dashboard.


2.2 Core User Journeys
Journey 1: New Broker Onboarding & First Shipment Entry
1. Broker visits the PortKey web application and lands on the secure login portal.
2. Click "Sign Up" to create a professional broker account.
3. Navigates to "Create New Shipment" from the sidebar.
4. Encodes Shipment Details: Manually enters the Bill of Lading (BL) number, Consignee name, Port of Origin, and Container details.
5. Saves Record: The shipment is assigned a "Pending" status and appears in the active tracking list.
Journey 2: Shipment Lifecycle & Status Tracking
1. Broker logs in and is greeted by the Shipment Analysis Dashboard.
2. Filters Shipments: Uses the search bar to find a specific BL number or filters by "Pending Customs" to see what needs attention.
3. Updates Status: As the shipment arrives at the port, the broker clicks the record and updates the status from "In Transit" to "Arrived at Port."
4. Logs Milestones: The broker enters the "Date of Release" once customs clearance is granted.
5. Historical Archive: Marks the shipment as "Completed" once delivered, which moves it from the active list to the searchable archives.
Journey 3: Administrator Product Management
1. Admin logs in with specialized administrative credentials.
2. Global Overview: Accesses a master view of all shipments encoded by all clerks/brokers in the system.
3. Performance Analysis: Views the "Shipment Analysis" module to track KPIs, such as the total volume of shipments cleared per month or average clearance lead times.
4. Record Auditing: Reviews historical shipment data and attached documents for compliance checks.
5. User Management: Manages broker accounts, ensuring only authorized personnel can access the sensitive logistical data.
2.3 Feature List (MoSCoW)
MUST HAVE
1. User Authentication: Secure register, login, and logout for Brokers.
2. Shipment Management (CRUD): Ability to create, view, update, and delete shipment records manually.
3. Role-Based Access Control (RBAC): Row-level security so Brokers see their specific shipments, while Admins see everything.
4. Shipment Lifecycle Status: A field to update statuses (e.g., In Transit, Arrived at Port, Customs Processing, Released).
5. Basic Shipment Search: Ability to find a record by Bill of Lading (BL) number or Consignee.
6. Admin Dashboard: Oversight of all user accounts and global shipment volume.
SHOULD HAVE
1. Shipment Analysis Module: A simple UI dashboard showing count of active vs. completed shipments.
2. Input Validation: Form validation to ensure BL numbers and dates are entered correctly to prevent data errors
3. Timestamp Logging: Automatically tracking when a shipment status was last changed.
COULD HAVE
1. Document Upload: Ability to attach a PDF or Image (Invoice/Packing List) to a specific shipment record.
2. Shipment "Flagging": Ability to mark a shipment as "Urgent" or "Delayed" with a red visual indicator.
3. Export to CSV/Excel: One-click download of the shipment list for monthly reporting.
4. Client Management: A separate table to store frequent Consignee/Client details to avoid re-typing them.
WON'T HAVE
1. AI Data Extraction: Automated parsing of documents (we are keeping it manual for the MVP).
2. Real-time GPS Tracking: Integration with physical truck or vessel tracking hardware.
3. Payment Gateway: Processing of customs duties or brokerage fees directly in-app.
4. Email/SMS Notifications: Automated alerts sent to clients upon clearance.
5. Multi-language Support: The system will be English-only for the initial release.

2.5 Acceptance Criteria
AC-1: Successful Broker Registration & Onboarding
* Given I am a new customs broker.
* When I enter a valid email, name, and strong password.
* And I click "Create Account".
* Then my account should be created in the database with ROLE_USER.
* And I should be redirected to the Shipment Dashboard (empty state).
AC-2: Manual Shipment Lifecycle (The "Core Loop")
* Given I am logged in as a Broker.
* When I fill out the "New Shipment" form with a Bill of Lading (BL), Consignee, and ETA.
* And I click "Save Shipment".
* Then the record should appear in my Active Shipments table.
* And I should be able to update the status to "Arrived at Port".
* And the Shipment Analysis card should increment the "Active" count by 1.
AC-3: Data Isolation (Privacy Check)
* Given I am logged in as Broker A.
* When I attempt to access a shipment ID belonging to Broker B via direct URL.
* Then the system should return a 403 Forbidden or 404 Not Found.
* And Broker B’s shipments should never appear in Broker A’s search results or Analysis dashboard.
3.0 NON-FUNCTIONAL REQUIREMENTS
3.1 Performance Requirements
* API Response Time: =< 1.5 seconds for shipment queries (important for brokers managing hundreds of records).
* Web Load Time: =< 2.5 seconds for the Dashboard on a 4G/LTE connection (brokers often work in ports with varying signal).
* Data Integrity: 0% data loss during shipment status updates.
* Database Queries: Complex filters on the Shipment table must complete within $300ms
3.2 Security Requirements
* Transport Security: Mandatory HTTPS (TLS 1.2+) for all traffic (No HTTP -> HTTPS redirects causing 405s!).
* Authentication: Stateless JWT tokens stored in HttpOnly cookies.
* Password Policy: BCrypt hashing with a cost factor of 12.
* CORS Policy: Strict origin matching (Only allow portkey.vercel.app and localhost:3000).
* Environment Variables: No hardcoded URLs or secrets in the codebase.
3.3 Compatibility Requirements
* Browsers: Chrome, Safari, and Firefox (95% of broker workstations).
* Mobile Web: Must be fully functional on mobile browsers (Safari for iOS, Chrome for Android) for "on-the-dock" updates.
* Screen Sizes: Optimised for 13" Laptops (Standard broker hardware) and Mobile (375px+).
3.4 Usability Requirements
* Efficiency: A broker should be able to manually encode a 10-field shipment in under 60 seconds.
* Feedback: Success/Error "Toasts" must appear for every database action (Save/Update/Delete).
* Scannability: Status badges must be color-coded (e.g., Green for "Cleared", Amber for "Pending", Red for "Delayed").

6.0 DATABASE DESIGN
6.1 Entity Relationship Diagram
  

Detailed Relationships:
* One-to-Many: User → Shipments (A broker can create and manage multiple active shipments).
* One-to-Many: User → Payments (A user can have multiple payment records for Pro plan upgrades).
* One-to-Many: User → Devices (A user can log in on multiple Android devices, each with an FCM token).
* One-to-Many: Shipment → ShipmentItems (A single container shipment can contain multiple cargo items).
* One-to-Many: Shipment → Documents (A shipment can have multiple attached files like BL, CI, PL).
* Many-to-One: Shipments → User (Shipments are tied to the specific broker who created them).
Key Tables:
* users: Core user accounts, roles (BROKER/ADMIN), subscription plan, and authentication data.
* shipments: The central operational entity tracking vessel details, container info, doomsday dates, and clearance status.
* shipment_items: Individual cargo lines within a shipment, including declared values and calculated PHP duties.
* documents: Metadata and Cloudflare R2 file URLs for uploaded customs documents.
* payments: Audit log of PayMongo transactions for account upgrades.
* devices: Storage for Android FCM push notification tokens.
* notifications: (Optional/Audit) Log of automated demurrage warning and critical emails sent by the cron job.
Table Structure Summary:
* users: id, email, password_hash, full_name, role (BROKER/ADMIN), plan (FREE/PRO), google_id, created_at
* shipments: id, user_id, vessel_name, voyage_number, arrival_date, port_of_discharge, client_name, container_numbers, free_days, doomsday_date, status (ARRIVED/LODGED/ASSESSED/PAID/RELEASED), lane (GREEN/YELLOW/RED), entry_number, or_number, deleted_at, created_at
* shipment_items: id, shipment_id, description, hs_code, quantity, declared_value, currency, php_converted_value, exchange_rate, created_at
* documents: id, shipment_id, file_name, file_url, type (BL/CI/PL/PERMIT/OTHER), size_bytes, uploaded_at
* payments: id, user_id, amount, status, reference_number, created_at
* devices: id, user_id, fcm_token, device_model, registered_at

* M3 (End Week 12): Android XML application complete with push notifications.

Critical Path:
1. Spring Boot custom JWT Authentication system (Week 1-2).
2. Shipment CRUD and doomsdayDate logic (Week 3).
3. Next.js and Android integration blocked until API endpoints are tested (Month 2 & 3).
4. Cross-platform testing and timezone resolution (Week 13).
Risk Mitigation:
* Strict Sequencing: Do not start Next.js or Kotlin development until the Spring Boot backend is returning correct data.
* Timezone Complexity: Actively monitor and test the UTC vs UTC+8 PHT discrepancy for the 8 AM demurrage cron job early in the integration phase.
* Free-Tier Limits: Cache ExchangeRate-API data for 24 hours in the database to prevent hitting monthly request limits.
* Scope Control: Strictly adhere to the MoSCoW priorities, ensuring core constraints (No Jetpack Compose, No Clerk) are respected before addressing any "Bonus" features.
